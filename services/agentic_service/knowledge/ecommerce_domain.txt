## 1. Domain Overview

### 1.1 Definition / Explanation

An **e-commerce web application** is a digital platform that enables users to discover, evaluate, purchase, pay for, receive, return, and review products or services through the internet. E-commerce systems support business transactions between sellers and buyers, including **B2C**, **B2B**, **C2C**, **D2C**, subscription commerce, digital goods, marketplace commerce, and omnichannel retail.

A production-grade e-commerce application is not only a product listing website. It is a complete transaction system that includes product catalog management, inventory control, pricing, promotions, cart management, checkout, payment processing, order management, fulfillment, shipping, returns, refunds, analytics, security, compliance, customer service, and operational monitoring.

### Key Components

| Component          | Purpose                                                                     |
| ------------------ | --------------------------------------------------------------------------- |
| Product Catalog    | Stores product data, categories, variants, images, SKUs, attributes         |
| Search & Discovery | Helps users find products using keywords, filters, ranking, recommendations |
| Cart               | Temporarily stores user-selected items                                      |
| Checkout           | Collects shipping, billing, payment, and confirmation details               |
| Payment Processing | Handles card payments, wallets, bank transfers, refunds, disputes           |
| Order Management   | Tracks the full order lifecycle                                             |
| Inventory          | Tracks stock availability across warehouses and sales channels              |
| Fulfillment        | Handles picking, packing, shipping, delivery, and returns                   |
| Customer Accounts  | Stores profiles, addresses, preferences, order history                      |
| Admin Panel        | Allows business users to manage catalog, orders, users, promotions          |
| Analytics          | Tracks revenue, conversion, retention, customer behavior                    |
| Security           | Protects data, payments, authentication, APIs, and infrastructure           |

### Real-world Examples

| Platform Type         | Example                                     |
| --------------------- | ------------------------------------------- |
| B2C Retail            | Amazon, Walmart, Nike Store                 |
| Marketplace           | eBay, Etsy, AliExpress                      |
| D2C                   | Apple Store, Tesla Store                    |
| Digital Products      | Udemy, Steam, Gumroad                       |
| Subscription Commerce | Netflix, Spotify, SaaS billing portals      |
| B2B Commerce          | Alibaba, industrial supply ordering portals |
| Grocery Commerce      | Instacart, Uber Eats grocery                |
| Omnichannel Retail    | Online ordering with physical store pickup  |

### Design Considerations

A good e-commerce system must support:

* High availability during traffic spikes.
* Accurate inventory and pricing.
* Secure payment handling.
* Fast product search and filtering.
* Reliable order state transitions.
* Scalable catalog and image delivery.
* Fraud prevention.
* Clear audit trails.
* Graceful failure handling.
* Admin and operational visibility.

### Edge Cases

| Edge Case                                 | Example                                                   |
| ----------------------------------------- | --------------------------------------------------------- |
| Product sold out during checkout          | User adds item to cart, but another user buys final stock |
| Payment succeeds but order creation fails | Gateway charges card, backend crashes before order save   |
| Duplicate payment webhook                 | Payment provider sends same webhook multiple times        |
| Discount conflict                         | User applies coupon to already discounted bundle          |
| Address invalid                           | Shipping provider rejects delivery address                |
| Price changed in cart                     | Product price changes before checkout                     |
| Cart abandoned                            | User leaves without completing checkout                   |
| Guest checkout merge                      | Guest later logs in and has existing cart                 |
| Refund partial failure                    | Refund processed but internal order status not updated    |
| Marketplace seller unavailable            | Seller pauses store while products remain listed          |

### Best Practices

* Use clear domain boundaries: catalog, cart, checkout, payment, order, inventory, fulfillment.
* Maintain immutable order records after purchase.
* Use idempotency keys for payment and order APIs.
* Use event logs for critical business transitions.
* Validate prices and inventory at checkout, not only when adding to cart.
* Separate user-facing product data from internal inventory data.
* Use asynchronous processing for emails, invoices, notifications, fulfillment, analytics.
* Maintain audit logs for admin actions.
* Use secure defaults for authentication, authorization, payments, and APIs.

### Common Mistakes

* Treating the cart as a final order.
* Not locking or reserving inventory during checkout.
* Trusting frontend price values.
* Storing payment card data unnecessarily.
* Mixing product catalog logic with order logic.
* Not handling duplicate webhooks.
* Not supporting partial refunds or partial shipments.
* Designing only for happy-path checkout.
* Ignoring SEO, accessibility, and mobile performance.
* Building admin features as an afterthought.

## 2. Core Business Concepts

### 2.1 Commerce Models

#### Definition / Explanation

E-commerce systems vary based on the relationship between buyers and sellers.

| Model                 | Description                                       |
| --------------------- | ------------------------------------------------- |
| B2C                   | Business sells directly to consumers              |
| B2B                   | Business sells to other businesses                |
| C2C                   | Consumers sell to consumers                       |
| D2C                   | Manufacturer or brand sells directly to consumers |
| Marketplace           | Multiple sellers sell through one platform        |
| Subscription Commerce | Recurring billing for products or services        |
| Digital Commerce      | Sale of downloadable or licensed digital goods    |
| Omnichannel Commerce  | Integration of online and offline channels        |

### Key Components

| Concept     | Description                      |
| ----------- | -------------------------------- |
| Buyer       | Purchases products or services   |
| Seller      | Offers products or services      |
| Product     | Item or service being sold       |
| Catalog     | Organized collection of products |
| Cart        | Temporary purchase selection     |
| Checkout    | Transaction finalization flow    |
| Order       | Confirmed commercial transaction |
| Payment     | Monetary settlement              |
| Fulfillment | Delivery of goods or services    |
| Return      | Buyer sends item back            |
| Refund      | Money returned to buyer          |
| Promotion   | Discount or incentive mechanism  |

### Real-world Examples

* A clothing brand selling directly through its own website is **D2C**.
* A grocery delivery platform connecting stores and customers is **marketplace commerce**.
* A company buying office supplies through a negotiated-price portal is **B2B commerce**.
* A video course platform selling downloadable lessons is **digital commerce**.

### Technical Implementation Notes

Commerce model affects system design.

| Feature   | B2C                 | B2B                   | Marketplace             |
| --------- | ------------------- | --------------------- | ----------------------- |
| Pricing   | Public pricing      | Contract pricing      | Seller-specific pricing |
| Checkout  | Simple checkout     | Approval workflows    | Seller split orders     |
| Payments  | Card/wallet         | Invoice, credit terms | Split payouts           |
| Inventory | Owned stock         | Bulk stock            | Seller-managed stock    |
| Users     | Individual accounts | Organization accounts | Buyers and sellers      |
| Shipping  | Single merchant     | Bulk shipments        | Multi-seller shipments  |

### Design Considerations

* B2B systems need organization accounts, purchase orders, approval workflows, tax exemption, negotiated pricing, credit limits, and invoices.
* Marketplace systems need seller onboarding, seller verification, commission logic, payout schedules, dispute handling, seller ratings, and multi-party refunds.
* Subscription commerce needs recurring billing, plan upgrades, cancellations, retries, grace periods, and proration.
* Digital commerce needs license keys, download links, entitlement checks, DRM, and access expiration.

### Edge Cases

* Buyer belongs to multiple organizations.
* Seller has restricted shipping regions.
* Subscription renewal payment fails.
* Marketplace seller cancels after buyer payment.
* B2B customer exceeds credit limit.
* Digital product download link expires.
* Order contains products from multiple sellers.
* Tax differs for physical and digital products.

### Best Practices

* Model commerce type explicitly.
* Avoid hardcoding B2C assumptions.
* Support extensible pricing and fulfillment rules.
* Use configurable workflows for order approval and cancellation.
* Design data models for multi-party transactions if marketplace support is planned.

### Common Mistakes

* Building a B2C checkout and later forcing B2B features into it.
* Treating all products as physical goods.
* Ignoring seller-level policies in marketplaces.
* Not separating platform revenue from seller revenue.
* Not supporting organization-level permissions.

## 3. System Architecture

### 3.1 Architecture Overview

#### Definition / Explanation

E-commerce architecture defines how frontend applications, backend services, databases, integrations, queues, caches, and infrastructure work together to deliver a reliable commerce experience.

A basic architecture:

```text
                         +-------------------+
                         |  Customer Browser |
                         +---------+---------+
                                   |
                                   v
                         +-------------------+
                         | Frontend Web App  |
                         +---------+---------+
                                   |
                                   v
                         +-------------------+
                         | API Gateway / BFF |
                         +---------+---------+
                                   |
          +------------------------+------------------------+
          |                        |                        |
          v                        v                        v
+------------------+     +------------------+     +------------------+
| Catalog Service  |     | Checkout Service |     | Order Service    |
+--------+---------+     +--------+---------+     +--------+---------+
         |                        |                        |
         v                        v                        v
+------------------+     +------------------+     +------------------+
| Product Database |     | Payment Gateway  |     | Order Database   |
+------------------+     +------------------+     +------------------+

Shared Infrastructure:
Cache | Search Index | Message Queue | Object Storage | Monitoring | Logging
```

### Key Components

| Component        | Responsibility                         |
| ---------------- | -------------------------------------- |
| Frontend         | User interface and interaction         |
| API Gateway      | Routing, authentication, rate limiting |
| Backend Services | Business logic                         |
| Database         | Persistent data storage                |
| Cache            | Fast access to frequent data           |
| Search Index     | Product discovery                      |
| Message Queue    | Async processing                       |
| Object Storage   | Product images, invoices, documents    |
| CDN              | Fast static asset delivery             |
| Monitoring       | Health, logs, metrics, alerts          |
| External APIs    | Payment, shipping, tax, email, SMS     |

### Real-world Examples

* Small store: monolithic backend with PostgreSQL and Stripe.
* Medium store: modular monolith with Redis, queue, and search engine.
* Large marketplace: microservices with Kafka, distributed databases, seller services, payout services, fraud detection, and event sourcing.

### Technical Implementation Notes

#### Monolith

```text
Frontend
   |
Backend Application
   |-- Catalog Module
   |-- Cart Module
   |-- Checkout Module
   |-- Payment Module
   |-- Order Module
   |-- Admin Module
   |
Single Database
```

Best for early-stage systems, university projects, MVPs, and small teams.

#### Modular Monolith

```text
Single Deployable Backend
   |
Internal Domain Modules
   |-- catalog/
   |-- users/
   |-- cart/
   |-- checkout/
   |-- orders/
   |-- payments/
   |-- inventory/
   |-- fulfillment/
```

Good balance between simplicity and maintainability.

#### Microservices

```text
Catalog Service
Cart Service
Checkout Service
Payment Service
Order Service
Inventory Service
Shipping Service
Notification Service
Analytics Service
```

Best for large teams, high scale, independent deployments, and complex domains.

### Design Considerations

| Decision                  | Consideration                                     |
| ------------------------- | ------------------------------------------------- |
| Monolith vs Microservices | Team size, deployment complexity, domain maturity |
| SQL vs NoSQL              | Transactions, relationships, flexibility          |
| Sync vs Async             | Immediate response vs eventual processing         |
| Cache Strategy            | Speed vs freshness                                |
| Search Engine             | Product discovery complexity                      |
| Event-driven Design       | Decoupling and scalability                        |
| API Gateway               | Security and routing                              |
| Multi-region              | Availability and latency                          |

### Edge Cases

* Service timeout during checkout.
* Payment gateway unavailable.
* Search index out of sync with database.
* Cache contains stale price.
* Queue consumer fails repeatedly.
* Database transaction partially completes.
* Duplicate order events.
* Admin changes product while customer checks out.

### Best Practices

* Start with a modular monolith unless clear scale requires microservices.
* Keep checkout and payment logic highly reliable.
* Use database transactions for order creation.
* Use idempotency for all payment-sensitive APIs.
* Use queues for non-blocking operations.
* Use observability from the beginning.
* Use feature flags for risky changes.
* Use API versioning.

### Common Mistakes

* Starting with microservices too early.
* Sharing one database across many microservices without boundaries.
* Not documenting service responsibilities.
* Not implementing retry and compensation logic.
* Using cache as the source of truth.
* Ignoring distributed transaction complexity.

# 4.1 Product & Catalog Module

## Definition / Explanation

The product and catalog module manages all information needed to present products to customers and administrators. It includes product names, descriptions, images, categories, variants, attributes, SKUs, pricing, availability, brand data, SEO metadata, related products, product status, and merchandising rules.

A catalog is different from inventory. The catalog explains **what can be sold**. Inventory explains **how much stock is available**.

## Key Components

| Component        | Description                                     |
| ---------------- | ----------------------------------------------- |
| Product          | Main sellable item                              |
| SKU              | Unique stock keeping unit                       |
| Variant          | Sellable variation such as size/color           |
| Attribute        | Product characteristic such as material, weight |
| Category         | Product grouping                                |
| Brand            | Manufacturer or label                           |
| Collection       | Curated group of products                       |
| Product Media    | Images, videos, 3D assets                       |
| Product Status   | Draft, active, archived, out of stock           |
| SEO Metadata     | Slug, title, description                        |
| Related Products | Cross-sell and upsell items                     |

## Product Types

| Type                 | Description                      | Example                |
| -------------------- | -------------------------------- | ---------------------- |
| Physical Product     | Tangible item requiring shipping | Shoes, phones          |
| Digital Product      | Downloadable or accessible item  | E-book, course         |
| Subscription         | Recurring product or service     | Monthly box, SaaS plan |
| Bundle               | Group of products sold together  | Gift pack              |
| Configurable Product | Product built by user selections | Custom laptop          |
| Service Product      | Non-tangible service             | Consultation           |
| Gift Card            | Stored value product             | $50 voucher            |
| Pre-order Product    | Sold before availability         | New device launch      |

## Real-world Examples

A shoe product may have:

```text
Product: Running Shoe
Variants:
- SKU: SHOE-BLK-40, Color: Black, Size: 40
- SKU: SHOE-BLK-41, Color: Black, Size: 41
- SKU: SHOE-WHT-40, Color: White, Size: 40
```

A digital course may have:

```text
Product: React Masterclass
Type: Digital
Delivery: Account entitlement
No shipping required
Access period: Lifetime
```

## Design Considerations

* Product IDs should be stable and not based on names.
* SKU must be unique and meaningful for warehouse and operations.
* Product slugs should support SEO but must not be the primary identifier.
* Product price should be stored in minor currency units, such as cents.
* Product images should be served through a CDN.
* Product data should support draft and publish workflow.
* Large catalogs require indexing and caching.
* Variant modeling must support product-specific options.

## Edge Cases

| Edge Case                             | Handling                                      |
| ------------------------------------- | --------------------------------------------- |
| Product has no image                  | Show placeholder image                        |
| Product deleted after order           | Keep order snapshot                           |
| Variant discontinued                  | Hide from purchase but preserve history       |
| SKU duplicated                        | Enforce unique constraint                     |
| Product price changes during checkout | Recalculate price before payment              |
| Product has thousands of variants     | Use pagination and efficient option filtering |
| Digital product requires license      | Generate entitlement after payment            |
| Product belongs to many categories    | Use many-to-many relationship                 |

## Best Practices

* Store product snapshots inside orders.
* Separate admin catalog data from public product response.
* Use searchable normalized fields for filters.
* Use structured attributes for filtering.
* Validate product completeness before publishing.
* Keep media metadata such as alt text for accessibility.
* Track product version changes for auditability.

## Common Mistakes

* Treating product and variant as the same entity.
* Storing price only in frontend.
* Not supporting product drafts.
* Not preserving old product details in past orders.
* Hardcoding categories.
* Ignoring SEO metadata.
* Using floating-point numbers for prices.

# 4.2 User & Identity Module

## Definition / Explanation

The user and identity module manages customer accounts, authentication, authorization, profiles, addresses, preferences, sessions, guest checkout, account security, and user lifecycle events.

Authentication answers: **Who is the user?**
Authorization answers: **What is the user allowed to do?**

## Key Components

| Component        | Description                            |
| ---------------- | -------------------------------------- |
| User Account     | Identity record                        |
| Profile          | Name, phone, preferences               |
| Credentials      | Password, OAuth identity, SSO identity |
| Session          | Login state                            |
| Role             | Customer, admin, seller, support agent |
| Permission       | Specific action rights                 |
| Address Book     | Shipping and billing addresses         |
| Guest User       | Temporary unauthenticated buyer        |
| MFA              | Multi-factor authentication            |
| Account Recovery | Password reset and verification        |

## Authentication Methods

| Method          | Usage                              |
| --------------- | ---------------------------------- |
| Email/password  | Standard login                     |
| OAuth           | Login with Google, Facebook, Apple |
| JWT             | Stateless API authentication       |
| Session Cookies | Browser-based authentication       |
| SSO             | Enterprise or B2B login            |
| Magic Link      | Passwordless email login           |
| OTP             | Phone or email verification        |
| MFA             | Extra login security               |

## Authorization Models

| Model        | Explanation                    |
| ------------ | ------------------------------ |
| RBAC         | Role-Based Access Control      |
| ABAC         | Attribute-Based Access Control |
| ACL          | Access Control List            |
| Policy-based | Rules define permitted actions |

Example RBAC:

```text
Customer:
- View products
- Manage own cart
- View own orders

Admin:
- Manage catalog
- Manage orders
- Manage users

Seller:
- Manage own products
- View own sales
- Process own shipments
```

## Real-world Examples

* A customer logs in with Google OAuth and checks order history.
* A guest user checks out without creating an account.
* An admin updates a product price.
* A marketplace seller manages only their own products.
* A support agent can view orders but cannot issue refunds above a threshold.

* Use secure password hashing such as bcrypt, Argon2, or scrypt.
* Prefer HTTP-only secure cookies for browser sessions.
* Use JWT carefully; support refresh token rotation.
* Do not expose internal user IDs unnecessarily.
* Allow guest checkout but support account creation after purchase.
* Protect admin routes with stricter authorization.
* Use MFA for administrators.
* Track suspicious login behavior.
* Support account deletion and data export where legally required.

## Edge Cases

| Edge Case                                     | Handling                              |
| --------------------------------------------- | ------------------------------------- |
| User forgets password                         | Password reset flow                   |
| Email already registered through OAuth        | Link identity providers               |
| Guest checkout user later registers           | Merge order history                   |
| Duplicate accounts                            | Account linking workflow              |
| Token stolen                                  | Refresh token rotation and revocation |
| User changes email                            | Verify new email                      |
| Admin loses MFA device                        | Recovery process                      |
| Seller tries accessing another seller’s order | Enforce tenant ownership checks       |

## Best Practices

* Never store plaintext passwords.
* Use rate limiting on login and reset endpoints.
* Use email verification before sensitive actions.
* Use audit logging for admin actions.
* Apply least privilege permissions.
* Store refresh tokens securely.
* Validate ownership on every protected resource.
* Use centralized authorization middleware.

## Common Mistakes

* Trusting frontend role values.
* Returning too much user data in API responses.
* Not protecting admin APIs.
* Not rotating refresh tokens.
* Using weak password hashing.
* Forgetting guest-to-user migration.
* Not rate limiting login attempts.

# 4.3 Shopping Cart Module

## Definition / Explanation

The cart module stores products a user intends to buy before checkout. A cart may belong to an authenticated user, a guest session, or a device/browser. The cart is temporary, editable, and does not guarantee product price or stock until checkout validation.

## Key Components

| Component       | Description                         |
| --------------- | ----------------------------------- |
| Cart            | Container of cart items             |
| Cart Item       | Product variant and quantity        |
| Cart Owner      | User ID or guest session ID         |
| Cart Pricing    | Estimated subtotal, tax, discount   |
| Cart Expiration | Cleanup policy                      |
| Coupon          | Discount applied to cart            |
| Saved Cart      | Persistent cart for logged-in user  |
| Abandoned Cart  | Cart not checked out after a period |

## Real-world Examples

* A guest adds two shoes to cart and returns later using the same browser.
* A logged-in user adds products on mobile and sees them on desktop.
* A user applies a coupon and receives free shipping.
* A cart is abandoned and later targeted by email marketing.

## Design Considerations

* Cart prices are estimates and must be revalidated at checkout.
* Cart should support both guest and authenticated users.
* Cart merge logic must handle duplicate items.
* Cart should not permanently reserve inventory unless business rules require it.
* Carts should expire or be cleaned periodically.
* Cart APIs must validate product status, purchasability, and quantity limits.

## Workflow: Add Item to Cart

```text
1. User selects product variant.
2. Frontend sends variant ID and quantity.
3. Backend validates product exists and is active.
4. Backend validates quantity rules.
5. Backend retrieves current price.
6. Backend adds item or updates existing cart item.
7. Backend recalculates cart summary.
8. API returns updated cart.
```

## Edge Cases

| Edge Case                                | Handling                        |
| ---------------------------------------- | ------------------------------- |
| Same item added twice                    | Increase quantity               |
| Product becomes inactive                 | Show unavailable item           |
| Price changes                            | Recalculate cart                |
| Stock unavailable                        | Warn user before checkout       |
| Guest logs in                            | Merge guest cart with user cart |
| Coupon expires                           | Remove coupon and notify user   |
| Quantity exceeds limit                   | Reject or adjust quantity       |
| Cart contains digital and physical items | Split checkout requirements     |

## Best Practices

* Store cart server-side for reliability.
* Use session ID for guest carts.
* Recalculate cart totals on every meaningful change.
* Separate cart pricing from final order pricing.
* Track abandoned carts for analytics.
* Make cart operations idempotent where possible.

## Common Mistakes

* Treating cart price as final order price.
* Reserving inventory too early without expiration.
* Losing guest cart after login.
* Not handling deleted products in cart.
* Allowing negative or zero quantities.
* Storing cart only in browser local storage.

# 4.4 Checkout Module

## Definition / Explanation

Checkout is the process of converting a cart into a confirmed order. It collects customer information, shipping address, billing address, delivery method, tax calculation, payment method, and final confirmation.

Checkout is one of the most business-critical flows because it directly affects revenue, conversion rate, fraud exposure, payment correctness, and customer trust.

## Key Components

| Component        | Description                     |
| ---------------- | ------------------------------- |
| Checkout Session | Temporary state during checkout |
| Shipping Address | Destination for physical goods  |
| Billing Address  | Payment-related address         |
| Shipping Method  | Delivery option                 |
| Tax Calculation  | Region-based tax                |
| Payment Intent   | Payment authorization attempt   |
| Order Review     | Final validation                |
| Confirmation     | Order creation result           |

## Real-world Examples

* A user checks out with a card and chooses standard shipping.
* A user buys a digital product and skips shipping.
* A B2B user submits a purchase order for approval.
* A marketplace checkout splits products by seller.
* A user selects cash-on-delivery where supported.

```json
  "paymentMethodType": "card",
  "amounts": {
    "subtotal": 3998,
    "shipping": 500,
    "tax": 360,

  }
```

```http

POST /api/checkout/session/{checkoutId}/payment-intent
POST /api/checkout/session/{checkoutId}/confirm
```

## Workflow: Standard Checkout

```text
1. User opens checkout from cart.
2. System creates checkout session.
3. User enters shipping address.
4. System validates address.
5. System retrieves shipping options.
6. User selects shipping method.
7. System calculates tax, discount, shipping, and total.
8. User enters payment method.
9. System creates payment intent with payment provider.
10. Payment is authorized or captured.
11. System creates order using final cart snapshot.
12. System reduces or reserves inventory.
13. System sends confirmation email.
14. System redirects user to order confirmation page.
```

## Design Considerations

* Checkout must validate cart contents again.
* Payment must use backend-generated totals, not frontend values.
* Checkout should support idempotency.
* Checkout should handle payment authorization and capture models.
* Address validation can be internal or provider-based.
* Tax and shipping must be recalculated if address changes.
* Use clear error messages to reduce abandonment.
* Store checkout session expiration.

## Edge Cases

| Edge Case                                 | Handling                                             |
| ----------------------------------------- | ---------------------------------------------------- |
| Payment succeeds but order creation fails | Use reconciliation job and idempotent order creation |
| Cart changes during checkout              | Refresh checkout totals                              |
| Shipping unavailable                      | Ask user for another address                         |
| Payment requires 3D Secure                | Redirect or challenge user                           |
| User double-clicks confirm                | Idempotency key prevents duplicate order             |
| Inventory unavailable                     | Block checkout or suggest alternatives               |
| Tax provider unavailable                  | Retry or use fallback rule                           |
| Browser closes after payment              | Confirmation recoverable through order lookup        |

## Best Practices

* Use idempotency for checkout confirmation.
* Use a state machine for checkout progress.
* Keep checkout pages fast and minimal.
* Provide guest checkout.
* Avoid unnecessary account creation before purchase.
* Validate shipping and billing data server-side.
* Record payment provider references.
* Use webhooks to confirm payment states.

## Common Mistakes

* Trusting client-side totals.
* Creating orders before payment strategy is clear.
* Not handling payment webhooks.
* Not supporting retry after failed payment.
* Not validating inventory at final step.
* Having too many checkout form fields.

# 4.5 Order Management Module

The order management module manages confirmed purchases from creation through payment, fulfillment, shipment, delivery, cancellation, return, refund, and completion. Orders are legally and financially significant records and must preserve the state of the transaction at the time of purchase.

## Key Components

| Order              | Confirmed transaction      |
| Order Item         | Purchased product snapshot |
| Order Status       | Current lifecycle state    |
| Payment Status     | Payment state              |
| Fulfillment Status | Shipping/warehouse state   |
| Invoice            | Financial document         |
| Shipment           | Delivery package           |
| Return Request     | Customer return process    |
| Refund             | Money returned             |
| Audit Trail        | Historical changes         |

```text
Created

Payment Pending
  |
Paid
  |
Processing
  |
Fulfilled / Partially Fulfilled

Shipped
```

```text
Paid -> Refunded
```

## Best Practices

* Use state machines for status transitions.
* Store provider references for payments and shipping.
* Maintain order event history.
* Make fulfillment operations auditable.
* Notify users on major order changes.
* Reconcile orders against payment provider regularly.
* Use background jobs for post-order tasks.

## Common Mistakes

* Using one generic status for everything.
* Updating order prices after purchase.
* Deleting cancelled orders.
* Not handling partial operations.
* Not storing product snapshots.
* Not supporting admin audit trails.

# 4.6 Inventory & Supply Chain Module

## Definition / Explanation

Inventory management tracks available stock for sellable items across warehouses, stores, suppliers, and channels. Supply chain functionality includes procurement, receiving, stock movement, stock reservation, stock synchronization, backorders, safety stock, and warehouse operations.

## Key Components

| Component          | Description                           |
| ------------------ | ------------------------------------- |
| Stock Item         | Inventory record for SKU              |
| Warehouse          | Physical stock location               |
| Available Quantity | Sellable quantity                     |
| Reserved Quantity  | Quantity held for pending orders      |
| On-hand Quantity   | Physical stock count                  |
| Safety Stock       | Minimum buffer stock                  |
| Backorder          | Order accepted when stock unavailable |
| Stock Movement     | Increase/decrease/transfer event      |
| Reorder Point      | Threshold for restocking              |
| Supplier           | Source of goods                       |

## Real-world Examples

* Warehouse receives 100 units of SKU-A.
* Customer checkout reserves 2 units.
* Order cancellation releases reserved stock.
* Store pickup requires local branch inventory.
* Product is available for backorder if supplier restock is expected.
* Marketplace seller updates stock via seller dashboard.

## Technical Implementation Notes

### Inventory Quantity Formula

```text
available_quantity = on_hand_quantity - reserved_quantity - damaged_quantity
```

### Sample Inventory JSON

### Database Schema Example


## API Endpoint Examples


## Workflow: Inventory Reservation During Checkout

```text
1. Checkout confirmation starts.
2. System validates requested SKUs and quantities.
3. System checks available quantity.
4. System creates inventory reservation.
5. System processes payment.
6. If payment succeeds, reservation converts to committed stock deduction.
7. If payment fails, reservation is released.
8. If checkout expires, reservation is released by scheduled job.
```

## Design Considerations

* Inventory must be consistent during checkout.
* Use database locking or atomic updates to prevent overselling.
* Reservations should expire.
* Stock movement history is necessary for auditing.
* Multi-warehouse logic may select nearest stock source.
* Backorders require clear customer communication.
* Inventory updates from external systems should be idempotent.

## Edge Cases

| Edge Case                           | Handling                         |
| ----------------------------------- | -------------------------------- |
| Two users buy last item             | Atomic update or lock            |
| Payment fails after reservation     | Release reserved stock           |
| Warehouse stock mismatch            | Manual adjustment with audit log |
| Supplier delay                      | Notify backorder customers       |
| Split fulfillment                   | Ship from multiple warehouses    |
| Damaged stock                       | Reduce sellable inventory        |
| External ERP sends duplicate update | Idempotency key                  |
| Negative inventory                  | Block unless explicitly allowed  |

## Best Practices

* Keep inventory separate from product catalog.
* Use stock movement logs instead of silent quantity updates.
* Use atomic database operations.
* Use reservations for checkout.
* Monitor low-stock SKUs.
* Support inventory reconciliation.
* Display availability clearly to customers.

## Common Mistakes

* Using product table quantity field only.
* Not handling concurrent checkout.
* Not releasing expired reservations.
* Not tracking stock movement reason.
* Allowing overselling unintentionally.
* Not syncing marketplace/channel inventory.

* Use analytics to improve ranking.

* Track search queries with no results.
* Use faceted filtering.
* Reindex asynchronously after catalog updates.

* Searching directly from relational database at scale.
* Not indexing variant-level attributes.
* Not handling typos.
* Returning irrelevant products due to poor ranking.
* Not tracking search analytics.
* Showing unavailable products without explanation.

## Fraud Detection

### Fraud Signals

## Design Considerations

* Never store raw card numbers unless certified and required.
* Use payment provider-hosted fields or tokenization.
* Verify webhook signatures.
* Use idempotency keys.
* Store payment provider references.
* Support payment retry.
* Handle asynchronous payment methods.
* Support partial refunds.
* Use fraud scoring before fulfillment for risky orders.

## Edge Cases

| Edge Case                             | Handling                       |
| ------------------------------------- | ------------------------------ |

## Best Practices

* Use trusted payment gateways.
* Keep payment logic server-side.
* Store only tokens and references.
* Reconcile payments daily.
* Make refunds auditable.
* Separate payment status from order status.
* Use strong logging without sensitive payment data.
* Use 3D Secure where appropriate.

## Common Mistakes

* Trusting frontend amount.
* Not handling webhooks.
* Creating duplicate orders after payment retry.
* Not supporting asynchronous payments.
* Storing card data directly.
* Treating chargebacks like normal refunds.
* Not recording payment audit history.

| Time zone differences         | Store UTC and report local                    |
| Guest converts later          | Merge identity carefully                      |
| Bot traffic                   | Filter suspicious sessions                    |
| Partial refund                | Adjust item-level metrics                     |
| Marketplace order             | Report platform and seller revenue separately |

## Best Practices

* Define analytics taxonomy early.
* Track both frontend and backend events.
* Use dashboards for key business metrics.
* Use cohort analysis for retention.
* Use event IDs for deduplication.
* Protect personal data.
* Validate analytics accuracy against orders database.

## Common Mistakes

* Relying only on frontend analytics.
* Not tracking checkout failure reasons.
* Mixing gross and net revenue.
* Ignoring refunds in reports.
* No event naming standard.
* Running heavy reports on production database.

## 14. Edge Cases & Failure Handling

### 14.1 Failure Handling Overview

#### Definition / Explanation

E-commerce systems must handle failures gracefully because failures can lead to lost revenue, duplicate payments, overselling, customer frustration, fraud, or legal issues. Failure handling includes validation, retries, idempotency, compensation, reconciliation, fallback behavior, alerting, and manual recovery workflows.

### Key Components

| Component         | Description                          |
| ----------------- | ------------------------------------ |
| Idempotency       | Prevent duplicate operations         |
| Retry Logic       | Retry temporary failures             |
| Circuit Breaker   | Stop calling failing dependency      |
| Compensation      | Reverse partial actions              |
| Dead-letter Queue | Store failed async jobs              |
| Reconciliation    | Compare internal and external states |
| Fallback          | Alternative behavior                 |
| Alerting          | Notify operators                     |
| Audit Trail       | Trace what happened                  |
| Manual Review     | Human intervention                   |

## Common Failure Scenarios

| Scenario                            | Risk                           |
| ----------------------------------- | ------------------------------ |
| Payment success, order failure      | Customer charged without order |
| Order success, inventory failure    | Overselling                    |
| Inventory reserved, payment failure | Stock stuck reserved           |
| Shipping label failure              | Fulfillment delay              |
| Email failure                       | Customer not notified          |
| Search service down                 | Product discovery broken       |
| Tax service down                    | Incorrect totals               |
| Webhook duplicate                   | Duplicate refund/order update  |
| Admin mistake                       | Wrong price or stock           |
| Queue worker crash                  | Delayed processing             |

## Technical Implementation Notes

### Idempotency Record Example

```json
{
  "idempotencyKey": "checkout_123_confirm",
  "requestHash": "abc123",
  "status": "completed",
  "response": {
    "orderId": "ord_1001"
  },
  "createdAt": "2026-04-30T10:00:00Z"
}
```

### Compensation Example

```text
Checkout Confirmation:
1. Reserve inventory.
2. Capture payment.
3. Create order.
4. Send confirmation.

If step 3 fails after step 2:
- Retry order creation using payment reference.
- If still failing, create incident.
- Do not automatically refund unless business rule says so.
- Reconcile payment provider transaction.
```

## Design Considerations

* Not all failures should be retried immediately.
* Payment and order creation require special reconciliation.
* Use unique provider references.
* Use queue retries with backoff.
* Failed jobs should move to dead-letter queue.
* Critical business failures should alert humans.
* Design admin tools for manual correction.
* Maintain logs with request IDs.

## Edge Cases

| Edge Case                                    | Handling                               |
| -------------------------------------------- | -------------------------------------- |
| Duplicate checkout confirm                   | Return existing order                  |
| Webhook arrives before API response          | Use provider reference to update state |
| Order created twice                          | Unique checkout/order constraint       |
| Refund webhook delayed                       | Show refund pending                    |
| Inventory reservation expired during payment | Revalidate before capture              |
| Customer retries after timeout               | Idempotency returns previous result    |
| External service outage                      | Queue and retry                        |
| Admin changes price during checkout          | Checkout uses validated snapshot       |

## Best Practices

* Use idempotency on all money-moving operations.
* Use state machines for orders, payments, refunds, fulfillment.
* Use reconciliation jobs.
* Monitor failed checkout attempts.
* Use dead-letter queues.
* Log provider event IDs.
* Provide customer-safe error messages.
* Provide admin recovery tools.

## Common Mistakes

* Retrying payment capture without idempotency.
* Ignoring failed background jobs.
* No reconciliation with payment provider.
* No manual admin recovery path.
* Showing technical errors to customers.
* Not tracking partial failure state.

## 15. Legal & Regulatory Considerations

### 15.1 Legal Overview

#### Definition / Explanation

E-commerce platforms must comply with laws and regulations related to privacy, consumer protection, payments, taxes, accessibility, advertising, product safety, refunds, data retention, intellectual property, and cross-border trade.

Legal requirements vary by country, industry, product type, and customer location. Production systems should allow configuration of policies by region and product category.

### Key Components

| Area                 | Description                             |
| -------------------- | --------------------------------------- |
| Privacy Law          | Personal data collection and processing |
| Consumer Protection  | Fair pricing, returns, disclosures      |
| Tax Compliance       | Sales tax, VAT, GST                     |
| Payment Compliance   | PCI-DSS and payment rules               |
| Accessibility        | Equal access for disabled users         |
| Marketing Consent    | Email/SMS consent                       |
| Product Restrictions | Age-restricted or regulated products    |
| Refund Policy        | Clear return/refund rules               |
| Terms & Conditions   | Legal usage agreement                   |
| Cookie Consent       | Tracking disclosure and preference      |

## Real-world Examples

* EU users may require GDPR-compliant consent and data deletion rights.
* Some products cannot be shipped internationally.
* Digital goods may have different refund rules.
* Alcohol, medicine, weapons, and supplements may require stricter controls.
* Tax invoices may be legally required in some regions.
* Marketing emails require opt-in or unsubscribe mechanism.

## Technical Implementation Notes

### Consent Record Example

```json
{
  "userId": "user_123",
  "consentType": "marketing_email",
  "status": "granted",
  "timestamp": "2026-04-30T10:00:00Z",
  "source": "checkout_checkbox",
  "policyVersion": "privacy_policy_v3"
}
```

### Compliance Features

```text
- Privacy policy page
- Terms and conditions page
- Cookie banner
- Consent management
- Data export request
- Data deletion request
- Marketing unsubscribe
- Tax invoice generation
- Refund policy display
- Age verification where needed
```

## Design Considerations

* Store consent with timestamp and policy version.
* Support user data export.
* Support user deletion/anonymization where applicable.
* Keep invoices and order records for required retention periods.
* Show full price, tax, shipping, and fees before payment.
* Ensure refund and return policy is visible before purchase.
* Do not send marketing emails without proper consent.
* Validate restricted products by location and user age.

## Edge Cases

| Edge Case                                 | Handling                                         |
| ----------------------------------------- | ------------------------------------------------ |
| User requests deletion with order history | Anonymize personal data but retain legal records |
| Product banned in destination country     | Block checkout                                   |
| User withdraws marketing consent          | Stop marketing messages                          |
| Tax-exempt B2B customer                   | Store exemption certificate                      |
| Age-restricted product                    | Verify age before purchase                       |
| Cross-border return                       | Region-specific return rules                     |
| Cookie refusal                            | Disable non-essential tracking                   |
| Price displayed incorrectly               | Honor or correct based on policy and law         |

## Best Practices

* Consult legal experts for target regions.
* Make policies visible and understandable.
* Keep consent audit logs.
* Use region-specific compliance rules.
* Maintain tax calculation records.
* Provide clear refund and cancellation workflows.
* Implement accessibility standards.
* Keep legal content versioned.

## Common Mistakes

* Copying generic policies without matching actual behavior.
* Not storing consent evidence.
* Sending marketing emails after unsubscribe.
* Not supporting data deletion/export.
* Hiding fees until final checkout.
* Ignoring tax rules.
* Selling restricted products without checks.

### 17.2 Headless Commerce

#### Definition / Explanation

Headless commerce separates the frontend presentation layer from backend commerce services. The backend exposes APIs, while frontend experiences can be built independently for web, mobile, kiosks, social commerce, or IoT.

### Key Components

| Component        | Description                     |

## Real-world Examples

* A brand uses the same commerce backend for website, mobile app, and in-store kiosk.

## Design Considerations

* APIs must be stable and well-documented.
* Frontend teams need flexible data access.
* Performance depends on API design.
* SEO must be handled in frontend framework.
* Authentication must work across channels.

## Edge Cases

| Edge Case                    | Handling                    |

| SEO issue in SPA             | SSR or pre-rendering        |
| Channel-specific pricing     | Channel context in API      |
| Offline mobile usage         | Sync strategy               |

## Best Practices

* Use clear API contracts.
* Use BFF for frontend-specific needs.
* Support channel context.
* Use CDN and SSR where needed.
* Keep business rules backend-side.
* Use design systems for UI consistency.

## Common Mistakes

* Putting pricing rules in frontend.
* No API versioning.
* Over-fetching large payloads.
* Poor SEO implementation.
* No shared cart model.

```text

12. Always include currency in monetary records.
13. Always design for guest checkout and authenticated checkout.
```
